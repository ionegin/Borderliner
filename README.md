# PLAN: Миграция mister_notes с Cloud Run на Railway
Planner: Claude (Sonnet 5) | Executor: Gemini 3.5 Flash | Reviewer: отличная от executor модель (например GPT/DeepSeek — НЕ Gemini)
Репозиторий: github.com/ionegin/mister_notes, ветка main, HEAD на момент планирования: 8aba14a

## КОНТЕКСТ (не менять, только читать)

Бот на aiogram 3.10, сейчас работает через webhook + aiohttp-сервер (bot.py, функции on_startup/on_shutdown/main).
Известная проблема на Cloud Run: `on_shutdown()` вызывает `bot.delete_webhook()`, и при редеплое/скейле старый
инстанс гасит вебхук, который новый инстанс только что поставил — гонка, из-за которой бот "то работает, то нет".
Плюс in-memory state (pending_messages, user_locks, processing_users, user_requests) не переживает рестарт
и не шарится между несколькими инстансами.

Решение: перейти на long polling, убрать webhook-логику, задеплоить на Railway как один процесс (single instance,
без autoscale). Полностью убирать HTTP-сервер не обязательно, если Railway настроен на HTTP healthcheck — тогда
эндпоинт "/" держим, но НЕ как способ приёма апдейтов.

## ЦЕЛЬ (definition of done)

1. Бот запускается на Railway командой `python run.py`, использует `dp.start_polling(bot)`.
2. Webhook-код (on_startup/on_shutdown/aiohttp webhook_handler) удалён или переведён под флаг (см. Задачу 2).
3. `config.py` больше не содержит дублирующихся блоков LLM_REGISTRY / LLM_SMART.
4. Персистентность users.json вынесена за пределы эфемерной ФС контейнера (см. Задачу 4) — ИЛИ явно
   задокументирован Railway Volume как временное решение, если полная миграция на БД не входит в объём этой задачи.
5. Бот проходит smoke-test: /start, голосовое сообщение, кнопка "Сократи" — без ошибок в логах.
6. Ничего из функционала (голос, видео-кружочки, очередь/склейка сообщений, стили, перевод, /broadcast, /llm1 /llm2)
   не сломано.

## ЗАДАЧИ (выполнять по порядку, коммит на каждую)

### Задача 1 — Починить config.py (дубли)
Файл: `config.py`
- Убрать второй (нижний) блок `LLM_REGISTRY = {...}` и объединить оба реестра в один — сохранить ВСЕ модели
  из обоих блоков (llama70b, llama8b, gemma4-31b, gemma4-31b-free, gemini-flash, gemini-flash-lite, qwen-72b,
  gemma3-27b). Явно решить дубли по ключам, если конфликт значений — оставить более новую версию (gemini-2.0-flash
  из второго блока, а не потерять запись).
- Оставить ОДНО определение `LLM_CLEANUP` и ОДНО `LLM_SMART` в самом низу файла, с комментарием какой ключ
  за что отвечает.
- Удалить неиспользуемую `LLM_FAST` ИЛИ, если хочешь её оставить на будущее — прогрепать `ai_client.py` и
  `bot.py` и убедиться, что она нигде не должна использоваться вместо LLM_CLEANUP (сейчас не используется нигде).
- Hold-out тест (для ревьюера, НЕ для executor-а — не подглядывать): после правки `python -c "import config;
  print(config.LLM_CLEANUP, config.LLM_SMART, len(config.LLM_REGISTRY))"` должен вернуть валидные ключи
  из LLM_REGISTRY и итоговое количество моделей = объединению обоих старых реестров без потерь.

### Задача 2 — Убрать webhook, перейти на polling
Файл: `bot.py`
- Удалить `on_startup()`, `on_shutdown()`, `webhook_handler`, регистрацию `/webhook` роута.
- Убрать `bot.delete_webhook()` — ИЛИ, если хочешь подстраховаться от старого вебхука Cloud Run, оставить
  ОДИН вызов `await bot.delete_webhook(drop_pending_updates=True)` перед стартом polling (это безопасно —
  выполняется один раз при старте, а не при shutdown, гонки не создаёт).
- Новый `main()`:
  ```python
  async def main():
      await bot.delete_webhook(drop_pending_updates=True)  # на всякий случай, безопасно
      await dp.start_polling(bot)

  if __name__ == "__main__":
      asyncio.run(main())
  ```
- Health-check эндпоинт `/` — оставить ТОЛЬКО если Railway service настроен с HTTP healthcheck. Если оставляем,
  поднять его как параллельную aiohttp-задачу через `asyncio.gather(dp.start_polling(bot), run_health_server())`,
  не смешивая с приёмом апдейтов.
- Убрать `PORT`/`WEBHOOK_URL` из обязательных env — polling их не требует (кроме health-сервера, если он остался).

### Задача 3 — Dockerfile / Railway конфиг
Файлы: `Dockerfile`, (создать) `railway.toml` или `Procfile`
- CMD должен запускать `run.py`, не `bot.py` напрямую (в run.py уже настроено логирование до импорта bot —
  проверь, что порядок импортов не потерялся).
- Добавить `railway.toml`:
  ```toml
  [build]
  builder = "DOCKERFILE"

  [deploy]
  numReplicas = 1
  restartPolicyType = "ON_FAILURE"
  ```
  `numReplicas = 1` — критично, чтобы не поднялось два polling-инстанса одновременно (Telegram банит дублирующийся
  getUpdates 409-кой, но лучше не полагаться на это, а гарантировать один инстанс явно).
- Проверить `.dockerignore` — убрать упоминания файлов, не относящихся к текущему деплою (fly.toml и подобные
  остатки от других платформ, если они больше не нужны).

### Задача 4 — Персистентность users.json (минимальный вариант в рамках этой задачи)
Файл: `bot.py` (функции `cmd_start`, `cmd_broadcast`)
- Если используем Railway Volume: смонтировать volume на `/app/data`, убедиться что `TEMP_DIR` и `users.json`
  пишутся туда же (уже так, менять пути не нужно — просто задокументировать в README, что volume обязателен).
- НЕ переписывать хранение на БД в рамках этой задачи (это отдельный тикет) — просто убедиться, что без volume
  бот не потеряет пользователей молча, и оставить явный TODO-комментарий в коде + строчку в README.

### Задача 5 — Smoke-тест (executor выполняет сам, отчитывается в PR/коммите)
- Локально (или в Railway staging) прогнать: `/start`, отправить голосовое, дождаться расшифровки, нажать
  "Сократи", нажать "Изменить стиль" → любой стиль, отправить `/broadcast test` от НЕ-админ id (должен молча
  проигнорировать).
- Приложить лог старта — должно быть видно "Start polling" (или аналог из aiogram), НЕ "Webhook set to...".

## ЗАПРЕЩЕНО (guardrails для executor-а)
- Не трогать `core/keyboards.py`, `data/prompts.json`, `data/users.json` — не в объёме этой задачи.
- Не менять сигнатуры `transcribe_and_cleanup`, `add_to_queue`, `flush_queue` — вся логика очереди голосовых
  остаётся как есть, трогаем только транспорт (webhook→polling) и конфиг.
- Не удалять rate-limit/lock-логику под предлогом "она всё равно не нужна на одном инстансе" — она разумна
  и для одного инстанса тоже (защита от спама и двойных нажатий).
